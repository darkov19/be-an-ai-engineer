## 6. Challenges And Growth Areas

Charlie (Senior Developer): "Most of the rework clustered around boundary conditions: external APIs, async cleanup, transaction state, and browser stream state."

Dana (QA Engineer): "That is the right pattern to notice. The happy path was implemented, but reviews repeatedly found edge cases where the system could report success incorrectly or drift out of sync."

Elena (Junior Developer): "The CSV and scheduler paths also show that fallback behavior needs the same rigor as the primary path."

Recurring challenges:

- **Telemetry correctness:** Story 2.1 needed review patches for all-fail run status and ingestion accounting. This matters because Epic 3 and report generation will trust these metrics.
- **External API variability:** Workable summary/detail behavior, Personio domain fallback, HN search limitations, and brittle live company slugs all expose corpus reliability risk.
- **Async lifecycle cleanup:** Story 2.2 surfaced premature queue cleanup and disconnect handling risks in the SSE/task manager flow.
- **Frontend stream state:** Story 2.3 review patches flagged timeout races, EventSource error handling, and file input reset behavior.
- **Database transaction resilience:** CSV telemetry initially risked failure after transaction aborts; scheduler/report writes add more paths where partial failure must be explicit.
- **Architecture drift:** The architecture document mentions `sse-starlette`, while the story correctly required Starlette-native `StreamingResponse` to avoid a new dependency. This should be cleaned up to prevent future agents from reintroducing unnecessary packages.

---

## 7. Key Lessons

1. **Operational truth beats optimistic status.** If all sources fail, if a stream disconnects, or if telemetry writeback fails, the system must say so directly.
2. **Fallback paths are product paths.** CSV import, timeout banners, kill artifacts, and email alerts must be tested with the same seriousness as live ingestion.
3. **The corpus is now a dependency.** Epic 3 extraction quality will only be meaningful if Epic 2 ingestion quality is measurable and honest.
4. **Async state needs explicit terminal events.** Task lifecycle, queue cleanup, and frontend state must agree on completion and failure semantics.
5. **Review is finding real product risk.** The review patches were not cosmetic; they corrected status, security, validation, serialization, and persistence behavior.

## 9. Next Epic Preview

**Epic 3:** AI Signal Extraction & Interactive Evals (The "Prognosis Engine")

Epic 3 will extract structured signals from ingested jobs using the local Hermes proxy, validate LLM output with Pydantic schemas, manage a labeled eval set, render an eval dashboard, and enforce kill criteria based on corpus size and extraction accuracy.

Dependencies on Epic 2:

- `jobs.raw_text`, `jobs.url`, `jobs.source_slug`, and job status data must be reliable enough for extraction.
- `ingestion_runs` and weekly report telemetry must remain truthful because kill criteria and run summaries depend on them.
- Background task and SSE patterns can be reused for extraction/eval runs.
- CSV fallback should remain available if live source quality is insufficient.
- Scheduler and alerting paths now provide the operational frame for run summaries and failure notifications.

Preparation needed before Epic 3:

- Verify Hermes proxy health using explicit backend code and tests.
- Add extraction/eval schema migrations with non-conflicting version numbers.
- Decide how extraction data attaches to `jobs`: either new columns, related extraction tables, or both.
- Define the prompt versioning storage model before the first extraction script lands.
- Create or seed the hand-labeled eval dataset before evaluating extraction quality.
- Run a corpus sanity check: count jobs, per-source distribution, duplicate rate, empty `raw_text` rate, and AI relevance.
- Update architecture docs to match the implemented native SSE approach.
