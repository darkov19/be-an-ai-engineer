## 1. Executive Summary

Epic 4 delivered the Prognosis Engine: structured LLM extraction through the local Hermes boundary, auditable extraction storage, a hand-labeled eval dataset, precision/recall/F1 scoring, regression detection, an interactive `/evals` dashboard, and quality gates that lock or warn when corpus and extraction quality fall below product thresholds.

All four planned stories are complete in sprint status:

- Story 4.1: LLM Structured Extraction and Proxy Verification - Done
- Story 4.2: Labeled Eval Set Management and Accuracy Audits - Done
- Story 4.3: Interactive Evals Dashboard and Regression Charts - Done
- Story 4.4: Kill-Criterion Enforcement and Warning Banner - Done

The epic changed the project from an ingestion/discovery system into a measured AI signal pipeline. The key learning is that LLM extraction is only credible when every output is versioned, validated, scored, visualized, and blocked from corrupting downstream reports when quality drops.

## 6. Challenges and Growth Areas

Charlie (Senior Developer): "Most review findings were not about missing headline features. They were about terminal-state correctness: what happens when inputs are malformed, null, duplicated, partially valid, or below threshold."

Dana (QA Engineer): "That pattern is useful. Epic 5 should assume every analytical endpoint needs adversarial tests for empty data, stale summaries, partial rows, and bad state transitions."

Elena (Junior Developer): "Frontend and backend state had to agree on what locked, warning, and nominal mean. That kind of cross-layer semantic drift is easy to miss."

Recurring challenges:

- **Partial failure semantics required repeated tightening.** Story 4.1 review found that malformed, duplicate, or unknown model response IDs could abort partial valid batches before persistence.
- **Auditing mattered as much as success.** Batch-level response failures initially risked leaving selected rows pending and unaudited.
- **CLI and script execution needed production-like hardening.** Story 4.1 review patched extraction client failure handling and direct script imports.
- **Task/UI contracts needed exactness.** Story 4.3 review patched `useSSE` compliance, side-by-side audit layout, explicit-null request validation, and task-spawning test coverage.
- **Quality gate semantics were easy to misread.** Story 4.4 review found issues around latest-ingestion success, kill-before-render ordering, threshold combinations, artifact failure fallback, and full-page lock presentation.
- **Runtime artifacts can leak into source control.** Story 4.4 review flagged a generated kill artifact in the working tree.

---

## 7. Key Lessons

1. **LLM extraction needs a trust chain.** Validated corpus selection, Hermes health, strict schema validation, prompt versioning, eval scoring, and quality gates must all hold together.
2. **Partial success is a first-class path.** Batches can return fewer valid objects than requested; the system must persist valid rows and explicitly mark missing or invalid rows.
3. **Evaluation is product infrastructure.** The eval harness is not an internal test helper; it is the public craft signal and the input to downstream product enforcement.
4. **UI state must mirror backend semantics exactly.** If backend states are `locked`, `warning`, and `nominal`, frontend surfaces and tests need to assert those exact meanings.
5. **Quality gates must execute before report output is considered valid.** A locked run should not quietly generate usable report content.
6. **Generated artifacts need source-control discipline.** Run summaries and kill artifacts are valuable evidence, but accidental runtime files should be reviewed before committing.

## 9. Next Epic Preview

**Epic 5:** Geo-Segmented Job Intelligence and Search (The "Career Cockpit")

Epic 5 will compute and display market intelligence from the extracted corpus: geo-segmented skill rankings, co-occurrence clusters, salary-stack correlations, experience distributions, profile fit, skill-gap diffing, company stack fingerprints, static fallback pages, archive history, and shareable report images.

Dependencies on Epic 4:

- Extracted `skills`, `seniority`, `tech_stack`, `salary_band`, `remote_policy`, and `role_archetype` fields must be populated enough to support aggregation.
- Latest eval accuracy must be available and trustworthy before weekly reports render.
- Kill/warning system state must be respected before publishing market metrics.
- Run summaries should feed archive and public methodology artifacts.
- Dashboard and report views must distinguish missing data from meaningful market signals.

Preparation needed before Epic 5:

- Define analytical projections for geo segments, top skills, co-occurrence, salary-stack correlations, and experience thresholds.
- Establish what extraction coverage is sufficient for analytics beyond the global corpus-size threshold.
- Decide how `unknown` extraction values affect rankings and fit scores.
- Add tests for empty corpus, locked state, warning state, missing eval run, and sparse extraction coverage.
- Document final quality-gate semantics in planning artifacts if the current PRD/epic wording still conflicts.

## 12. Significant Discovery Check

No Epic 4 discovery invalidates Epic 5's product direction. The plan to build geo-segmented analytics and report surfaces is still sound.

Epic 4 did reveal three planning constraints Epic 5 should account for:

1. Analytical reports should be blocked or warned based on health state before users treat metrics as valid.
2. Sparse or `unknown` extraction values must be handled deliberately in rankings and profile diffing.
3. Public portfolio artifacts should explain methodology and limitations, not just show charts.
