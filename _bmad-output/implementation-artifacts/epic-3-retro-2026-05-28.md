## 1. Executive Summary

Epic 3 delivered the Company Radar: a validated company-signal registry, bounded canonical resolver, optional search and Wellfound providers, Common Crawl/YC/VC scale discovery, GitHub/Reddit long-tail providers, provider yield reporting, source freshness diagnostics, and API exposure for discovery health.

All four planned stories are complete in sprint status:

- Story 3.1: Company Signals and Canonical Source Resolver - Done
- Story 3.2: Vertex AI Search and Wellfound Direct Hiring Signal Providers - Done
- Story 3.3: Common Crawl, YC, and VC Scale Discovery Providers - Done
- Story 3.4: Long-Tail Signals and Provider Yield Reporting - Done

The epic changed source discovery from a small trusted-source registry into an evidence pipeline. The central learning is that discovery providers must be treated as noisy signal channels, not corpus truth. Epic 3 succeeded because every provider flows through normalization, bounded resolution, parser-backed validation, diagnostics, and reporting before anything becomes an active ingestion source.

## 5. What Went Well

Alice (Product Owner): "Epic 3 protects the product from a bad temptation: treating every discovery channel as a trusted job source. The trust boundary is much clearer now."

Charlie (Senior Developer): "The strongest technical choice was making provider output evidence-only until it survives canonical resolution and parser validation."

Key wins:

- Company discovery has a durable schema and API surface instead of scattered provider-specific logic.
- Direct ATS hits and company/domain signals use distinct paths, both with validation before activation.
- Resolver behavior stays bounded: no browser automation, no recursive crawling, no marketplace scraping.
- Optional providers are safe by default and report disabled/misconfigured states instead of silently disappearing.
- Reviews found material product risk and were resolved inside the epic.
- The latest discovery report shows actionable provider yield, not just raw counts.

## 6. Challenges And Growth Areas

Charlie (Senior Developer): "Most review findings came from the same family: provider edge cases around caps, malformed responses, redirects, diagnostics, and secret redaction."

Dana (QA Engineer): "That is expected for a provider-heavy epic, but it means future provider work should start from a shared checklist instead of rediscovering every boundary."

Elena (Junior Developer): "The provider code kept expanding in one service module. It is still workable, but the shape is getting harder to scan."

Recurring challenges:

- **Provider diagnostics were easy to lose.** Story 3.2 review found disabled, missing-credential, quota-exhausted, and API-failure diagnostics being overwritten or hidden.
- **External payloads needed stronger guards.** Vertex, Wellfound, Common Crawl, YC/VC, GitHub, and Reddit all surfaced malformed payload or response-size risks.
- **Quota and cap behavior required repeated correction.** Search provider quota reservation, Common Crawl record caps, Reddit cap diagnostics, and GitHub rate diagnostics all needed review attention.
- **Attribution was non-trivial.** Provider yield had to distinguish revalidation from newly activated sources and avoid double-counting company-signal results.
- **One deferred item remains from Story 3.1.** There is still no global per-run company signal cap; later providers added their own caps, but the orchestration layer still lacks a single aggregate guard.
- **Planning docs can lag implementation reality.** Story 3.2 replaced the legacy search plan with Vertex AI Search because the old API path was unavailable to new customers; the planning artifacts have now been corrected.

---

## 7. Key Lessons

1. **Evidence is not ingestion.** Provider hits become useful only after normalization, bounded resolution, validation, and diagnostics.
2. **Disabled providers are still product state.** Missing credentials, disabled flags, quota exhaustion, and misconfiguration must appear in reports.
3. **Provider yield is the decision surface.** Raw signal volume is less important than resolved sources, validated active sources, rejection reasons, and repeated noise.
4. **Review patterns should become provider standards.** The same checks recurred across providers: caps, redaction, malformed payloads, response-size limits, redirects, and failure isolation.
5. **The next epic depends on corpus credibility.** LLM extraction and evals will only be meaningful if active sources remain validated, fresh, and explainable.

## 9. Next Epic Preview

**Epic 4:** AI Signal Extraction & Interactive Evals (The "Prognosis Engine")

Epic 4 will extract structured fields from job descriptions through the local Hermes proxy, validate LLM output with Pydantic schemas, manage a labeled eval set, compute precision/recall, surface regression charts, and enforce kill/warning criteria based on corpus size and extraction quality.

Dependencies on Epic 3:

- Active `job_sources` must stay validation-gated.
- Provider yield should guide which channels are trusted for corpus expansion.
- Source freshness and repeated rejection diagnostics should be checked before extraction runs.
- Company signals remain evidence only; extraction should use ingested job corpus data, not raw provider snippets.
- Discovery reports should feed kill-criterion and run-summary artifacts.

Preparation needed before Epic 4:

- Rename planned eval/extraction migrations to avoid `V006`.
- Verify Hermes proxy health path and failure messages before extraction batches run.
- Define extraction storage so prompt version, schema version, validation status, and extraction errors are auditable.
- Lock the labeled eval seed and rubric before prompt tuning.
- Run a fresh corpus sanity check and source-discovery report before measuring extraction accuracy.
- Decide how provider yield affects corpus selection for initial extraction.
