## 6. Challenges and Growth Areas

Recurring challenges:

- **Generated content safety kept reappearing.** Story 5.4 found unescaped database/LLM content in static fallback HTML, and Story 5.5 carried that lesson into report rendering.
- **Public and offline surfaces needed stricter asset discipline.** Review found remote Google Fonts in an offline fallback and later required lightweight report pages with no remote fonts or CDN dependencies.
- **URL and slug handling required repeated tightening.** Story 5.4 found unsanitized fallback navigation, unencoded route fetches, and `re.match` slug validation that could accept trailing newline-style inputs.
- **Interactive UI paths needed deeper accessibility coverage.** Story 5.2 and Story 5.3 review findings covered tooltip clickability, keyboard anomaly resolution, live-region semantics, skill-specific button names, and reduced-motion behavior.
- **Static publishing semantics were easy to get subtly wrong.** Story 5.5 review found missing artifact pushes, stale deployment metadata, relative archive links, and health checks that could verify an old report instead of the current one.
- **Tests initially focused on headline rendering before review pushed them into edge paths.** Several patches added assertions for hover, resolve, keyboard, reduced motion, stale-profile fixed dates, delta colors, and publish workflow correctness.

Winston (Architect): "The pattern is not that the team forgot security or accessibility. The pattern is that every new public surface multiplies the places those concerns must be enforced."

Dana (QA Engineer): "For Epic 6, the same lesson applies to accountability data. Anything public, user-entered, or generated needs escaping and evidence links by default."

## 7. Key Lessons

1. **Analytics must consume quality state, not only data rows.** Story 5.1 was strongest because it kept locked/warning/nominal state in the analytics response contract.
2. **Static public output is product surface, not generated byproduct.** Report, archive, OG image, and cached fingerprint files need tests, escaping, deterministic generation, and documented public intent.
3. **Precompute live-demo paths.** Company fingerprint views should read cached projections and static fallbacks rather than triggering extraction or LLM work during screen-share moments.
4. **Accessibility is a workflow, not a final CSS pass.** Hover, keyboard, live updates, color labels, reduced motion, and repeated buttons all need acceptance-test-level coverage.
5. **Publish pipelines need idempotent artifact semantics.** A report is not published until generated assets, metadata, deployment output, archive links, and health checks refer to the same report.
6. **Review findings were concentrated in boundary behavior.** The epic shipped faster once tests covered boundary paths instead of only nominal rendering.
